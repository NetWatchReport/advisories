# NWTA-2026-002: gitmc-org-mcp-scanner

**NetWatch Threat Advisory**
**Published:** 2026-03-11
**Last Updated:** 2026-03-11
**TLP:** WHITE (unrestricted)
**Confidence:** HIGH

---

## Summary

NetWatch has identified a previously undocumented scanning tool designated **gitmc-org-mcp-scanner** that enumerates exposed Model Context Protocol (MCP) servers across the public internet. The tool operates in two distinct phases: web reconnaissance using a spoofed BitSightBot user agent with IP rotation across a /24 subnet, followed by targeted MCP protocol probing using python-httpx. The tool tests both MCP transport mechanisms (SSE and Streamable HTTP) to maximize server discovery.

The tool self-identifies as `gitmc-org-mcp-scanner` version 1.0.0 in its MCP initialization handshake. No public repository, documentation, or prior reporting exists for this tool. This is the **first known public documentation**.

## Indicators of Compromise

### User Agents

The tool uses different user agents for each scan phase:

```
Phase 1 (recon):  Mozilla/5.0 AppleWebKit/537.36 (KHTML, like Gecko) Chrome/123.0.6312.86 Safari/537.36 BitSightBot/1.0
Phase 2 (MCP):    python-httpx/0.28.1
```

The Phase 1 UA impersonates BitSight Technologies' legitimate security ratings crawler (Groma). BitSight does not publish their scanning IP ranges, making verification difficult. The Phase 2 UA is the default python-httpx string with no additional spoofing.

**Note:** The `BitSightBot/1.0` UA alone is not sufficient for detection — BitSight operates a legitimate scanning service. Correlation with Phase 2 behavior or the header fingerprints below is required to distinguish this tool from real BitSight traffic.

### MCP Client Identification

The tool self-identifies in the JSON-RPC `initialize` handshake:

```json
{
  "clientInfo": {
    "name": "gitmc-org-mcp-scanner",
    "version": "1.0.0"
  }
}
```

This is the **most reliable detection indicator** — match on the string `gitmc-org-mcp-scanner` in POST request bodies to MCP-associated paths (`/mcp`, `/messages`, `/sse`).

### HTTP Header Order Fingerprints

The tool produces two stable header orderings, one per phase:

```
Phase 2 POST hash (SHA-256, first 16 hex): 9dd0446ed1d11fe2
Phase 2 GET hash  (SHA-256, first 16 hex): fd1d47cebeb49965
```

Detection rule: compute SHA-256 of the pipe-delimited HTTP header key names (in received order), truncate to 16 hex characters, and compare against the hashes above.

**Note:** The Phase 1 recon hash (`978aba6751832856`) is a generic 4-header ordering (`Host|User-Agent|Accept|Accept-Encoding`) shared by dozens of scanning tools and is not useful for fingerprinting.

### MCP Protocol Version

The tool requests MCP protocol version `2025-06-18` (current specification) with capabilities:
- `sampling` — request model completions from the server
- `elicitation` — request structured input from users
- `roots` (with `listChanged: true`) — access filesystem roots

This capability set indicates intent to fully interact with discovered MCP servers, not merely confirm their existence.

### Target Endpoints

**Phase 1 — Web Reconnaissance:**
- `GET /` — baseline response
- `GET /api/v2/` — API version discovery
- `GET /ms/1.0/` — Microsoft/generic API discovery
- `GET /favicon.ico` — server fingerprinting

**Phase 2 — MCP Protocol Probing:**
- `POST /mcp` — Streamable HTTP transport (primary)
- `GET /sse` — SSE transport (legacy)
- `POST /messages` — Streamable HTTP transport (alternate path)

### Target Ports

Observed scanning port 8001 (LangServe / vLLM alternate). Additional ports are expected as more sightings are collected.

## Behavior Profile

### Two-Phase Scan Architecture

Unlike single-tool scanners, gitmc-org-mcp-scanner employs a deliberate two-phase approach:

**Phase 1 (Recon — 15 seconds):** Four IPs rotate through generic web discovery paths using a spoofed BitSightBot UA. This phase identifies live HTTP servers and gathers baseline response data. The BitSightBot impersonation may be designed to avoid WAF rules that whitelist known security rating services.

**Phase 2 (MCP Probe — 1 second):** A single IP switches to python-httpx and sends MCP `initialize` handshakes via all three known MCP transport entry points (`/mcp`, `/sse`, `/messages`). This ensures detection of MCP servers regardless of which transport they implement. The three probes complete within 1 second — a tight, automated burst.

### IP Rotation Strategy

Phase 1 distributes one request per IP across the /24 subnet (.38 → .40 → .38 → .39). Phase 2 concentrates on a single IP (.37). This asymmetry suggests the recon phase is designed to stay below per-IP rate limits while the MCP phase prioritizes speed over stealth.

### Scan Depth

The observed scan stopped at `initialize` — no `tools/list` or `tools/call` requests were sent. This indicates the tool is currently operating in **discovery mode**, cataloging MCP-capable servers without exploiting them. Whether a follow-up exploitation phase exists (from the same or different infrastructure) is unknown.

## Context

MCP server scanning represents an emerging threat category in 2026:

- **Enkrypt AI** (February 2026): Scanned 1,000 MCP servers and found 33% had critical vulnerabilities including tool poisoning, cross-origin escalation, and server-side request forgery.
- **8,000+ exposed MCP servers** cataloged in the wild as of February 2026.
- **Backdoored MCP servers** discovered on package registries (e.g., `postmark-mcp-server` silently exfiltrating email content).

Exposed MCP servers can allow attackers to execute arbitrary tool calls, access databases, read/write files, and exfiltrate credentials through the server's tool interface. Discovery scanning is the first stage of this attack chain.

This tool differs from NWTA-2026-001 (LLM-Scanner/2.0-Fast) in both targeting and methodology. LLM-Scanner targets inference API endpoints (Ollama, llama.cpp, OpenAI-compatible). gitmc-org-mcp-scanner targets the MCP control plane — the tool orchestration layer that sits above inference. Compromise of an MCP server is typically higher impact than compromise of a bare inference endpoint, as MCP servers often have filesystem access, database connections, and credential stores.

## Detection Guidance

### Recommended Detection Rules

**Priority 1 — Body content match (most reliable):**
Match on string `gitmc-org-mcp-scanner` in POST request bodies. Zero false-positive risk.

**Priority 2 — Header fingerprint:**
Match on header order hashes `9dd0446ed1d11fe2` or `fd1d47cebeb49965`.

**Priority 3 — Behavioral signature:**
Alert on requests to 2+ of the following paths from a single source IP within a 10-second window:
`/mcp`, `/sse`, `/messages`

Combined with `Content-Type: application/json` and a POST body containing `"method":"initialize"`.

### MCP Server Hardening

1. **Do not expose** MCP servers on public interfaces. MCP is designed for local tool orchestration, not internet-facing deployment.
2. **Require authentication** on all MCP transports (SSE and Streamable HTTP).
3. **Validate** MCP client identities in the `initialize` handshake before returning server capabilities or tool listings.
4. **Monitor** for `initialize` requests from unknown clients as an early warning of reconnaissance.

## Recommendations

1. **Block** the identified IPs and monitor the 185.180.141.0/24 range for continued activity
2. **Deploy** body-content detection rules for `gitmc-org-mcp-scanner` at your reverse proxy / WAF
3. **Audit** internet-facing MCP server deployments for authentication requirements
4. **Report** sightings of this tool to NetWatch (abuse@netwatch.report) to expand the fingerprint database
