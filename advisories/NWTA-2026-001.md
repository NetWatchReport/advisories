# NWTA-2026-001: LLM-Scanner/2.0-Fast

**NetWatch Threat Advisory**
**Published:** 2026-03-11
**Last Updated:** 2026-03-15
**TLP:** WHITE (unrestricted)
**Confidence:** HIGH

---

## Summary

NetWatch has identified a previously undocumented scanning tool designated **LLM-Scanner/2.0-Fast** that systematically enumerates exposed Large Language Model (LLM) inference endpoints across the public internet. The tool probes framework-specific API paths for Ollama, llama.cpp, OpenAI-compatible, and LM Studio servers. It has been observed operating from privacy VPN infrastructure (Mullvad), and multiple cloud compute providers (AWS EC2, Vultr, DigitalOcean) across 4 countries, with evidence of active user agent spoofing to evade detection.

This is the **first known public documentation** of this tool.

## Indicators of Compromise

### User Agents (actively evolving — do not rely on UA alone)

The tool's UA spoofing was observed evolving within a single scan session:

```
Stage 1 (primary):   LLM-Scanner/2.0-Fast
Stage 2 (stripped):  python-requests/2.31.0
Stage 3 (spoofed):   Mozilla/5.0 (Windows NT 6.2;en-US) AppleWebKit/537.32.36 (KHTML, live Gecko) Chrome/{randomized}.0.{randomized}.{randomized} Safari/537.32
```

Stages 1→2 occurred within the same scan session (the operator disabled their custom UA mid-scan). Stage 3 appeared ~8.5 hours later from different infrastructure, with randomized Chrome version numbers across instances (observed: 57.x, 60.x). This represents an operator actively implementing UA spoofing — future scans may use more convincing UAs.

**Spoofed UA tell:** The Stage 3 UA contains `live Gecko` instead of the standard `like Gecko` — a typo baked into the tool's spoofing template. This persists across all spoofed instances and Chrome version randomizations.

### HTTP Header Order Fingerprint

The tool produces a **stable HTTP header ordering** across infrastructure changes and UA spoofing. This is the most reliable detection indicator:

```
Header order hash (SHA-256, first 16 hex): a6e546c711ea2493
```

Detection rule: compute SHA-256 of the pipe-delimited HTTP header key names (in received order), truncate to 16 hex characters, and compare against the hash above.

### Target Endpoints

The tool probes the following paths on LLM-associated ports:

**Ollama API:**
- `GET /api/tags`
- `GET /api/version`
- `GET /api/generate`
- `GET /api/ps`
- `POST /api/show` (with model name from `/api/tags` response)

**llama.cpp / llama-server:**
- `GET /completion`
- `GET /slots`
- `GET /props`

**OpenAI-compatible:**
- `GET /v1/models`
- `GET /v1/completions`

**Generic health checks:**
- `GET /health`
- `GET /api/health`
- `GET /`

### Target Ports

Observed scanning the following LLM-associated ports:
- 11434 (Ollama default)
- 8080 (llama.cpp / vLLM / common proxy)
- 1234 (LM Studio default)

- 8001 (LangServe / vLLM alternate)

## Behavior Profile

### Scan Pattern

The tool performs a comprehensive, framework-agnostic enumeration of each target port. Rather than probing for a single framework, it sends the full endpoint set for Ollama, llama.cpp, and OpenAI-compatible APIs against every port. This allows it to identify any LLM framework on any port, regardless of whether the framework uses its default port.

Observed scan characteristics:
- **Burst mode:** 35 requests across 4 ports in ~22 seconds
- **Sequential port scanning:** Ports are scanned in ascending numerical order
- **Request repetition:** Some endpoints are probed multiple times per port (particularly `/health` and `/v1/models`)
- **Framework-agnostic:** Same probe set applied to all ports

### Infrastructure Rotation

The operator employs disposable cloud infrastructure with rapid provider rotation:

- **Mullvad VPN** for comprehensive burst scans (35 requests across 4 ports in 22 seconds)
- **AWS EC2** for distributed single-port validation probes (one request per IP, rotating instances within the same us-east-1 subnet)
- **GCP, Vultr, DigitalOcean** for continued scanning after abuse reports forced migration from AWS

The operator has been observed migrating providers within days of abuse reports: AWS (reported 2026-03-11, action confirmed 2026-03-13) → GCP (2026-03-12) → Vultr Mexico (2026-03-14) → DigitalOcean Frankfurt (2026-03-15). This demonstrates operational awareness of abuse reporting timelines and willingness to burn through cloud accounts.

Recent sightings show reduced scan depth — single `/api/tags` requests per IP rather than the full multi-port sweep seen in early Mullvad sessions. This may indicate a shift from discovery to targeted validation of previously catalogued endpoints.

UA spoofing was observed developing in real time during the initial Mullvad session: the tool's real name was used for the first 33 requests, stripped to the default `python-requests` UA for the final 2 requests of the same session, and replaced with spoofed Chrome strings (randomized version numbers, persistent `live Gecko` typo) on subsequent AWS instances. More recent sightings from Vultr and DigitalOcean have dropped the spoofed Chrome UA entirely, reverting to bare `python-requests`. All sessions share an identical HTTP header order fingerprint regardless of UA or source IP.

### Version Numbering

The `-Fast` suffix in the tool name implies the existence of other scan modes (potentially `-Deep`, `-Full`, or similar). The `2.0` version number implies a `1.x` predecessor with iteration history. This suggests an actively developed tool, not a one-off script.

## Context

This tool is consistent with the "Scanner" tier of the LLMjacking pipeline described in:

- **Pillar Security, "Operation Bizarre Bazaar"** (2025): Documents a three-tier pipeline (Scanner → Validator → Marketplace) used to industrialize compute theft from exposed LLM endpoints.
- **GreyNoise, "Ollama Honeypot Report"** (January 2026): Reports 91K+ attack sessions against Ollama honeypots with Nuclei-like tooling across 62 source IPs.

Neither report documents "LLM-Scanner" by name or provides matching fingerprints.

## Detection Guidance

### Recommended Detection Rules

**Priority 1 — Header fingerprint (most reliable):**
Match on header order hash `a6e546c711ea2493`. This persists across UA spoofing and IP rotation.

**Priority 2 — User agent string:**
Match on substring `LLM-Scanner` in the User-Agent header. Note: the operator has been observed spoofing this field.

**Priority 3 — Behavioral signature:**
Alert on requests to 3+ of the following paths from a single source IP within a 60-second window:
`/api/tags`, `/api/version`, `/completion`, `/slots`, `/props`, `/v1/models`, `/v1/completions`

### Spoofed UA Detection

The known spoofed UA contains `live Gecko` instead of the standard `like Gecko`. This typo can be used as an additional detection signal, though it may be corrected in future versions.

## Recommendations

1. **Block or alert** on the header order hash and known UAs at your reverse proxy / WAF
2. **Do not expose** LLM inference servers (Ollama, LM Studio, llama.cpp, vLLM, text-generation-webui) on public interfaces without authentication
3. **Monitor** for follow-up activity from different IPs after a scan — the scanner-to-validator pipeline typically involves a second-stage actor interacting with discovered endpoints
4. **Report** sightings of this tool to your threat intelligence provider and to NetWatch (abuse@netwatch.report) so we can expand the fingerprint database
